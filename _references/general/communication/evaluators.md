# EVL — Evaluators

> CTOs, tech leads, and engineering managers assessing whether to adopt the framework.

## Core Question

"Should we adopt this? What's the ROI?"

## Roles

- CTO / VP of Engineering
- Tech lead / Staff engineer
- Engineering manager
- Architecture review board member
- Developer experience lead

## Communication Content

### Essential (all Evaluators need)

- **Executive Overview**: What the framework is, what problem it solves, and how it differs from ad-hoc AI-assisted development. One-page summary with key differentiators and constraints.
- **Adoption Path**: Concrete steps from first contact to team-wide rollout. Prerequisites, time investment, and what changes in daily workflow.
- **Architecture Overview**: How the framework is structured — skills, review perspectives, onboarding layers, _references. Diagram of component relationships and extension points.

### Deep-dive (load for thorough evaluation or when Evaluator is the primary audience)

- **Comparison Guide**: How the framework compares to alternative approaches — raw prompting, other frameworks, no framework. Honest assessment of where the framework adds value and where it adds overhead.
- **Proof of Value**: Concrete before/after examples. Metrics from real usage: review coverage, onboarding time, consistency of AI output. Template for running a team pilot.
- **Migration Path**: How to introduce the framework into an existing project with established conventions. What to expect during transition, how to handle resistance, rollback strategy.
- **Cost Analysis**: Time investment for setup, learning curve, ongoing maintenance. What the framework costs in terms of token usage, context window, and developer attention.
- **Framework Value Proposition**: What the framework delivers in business terms -- faster onboarding, more consistent code quality, reduced rework. One-page summary for non-technical evaluators who need to justify the investment.
- **Framework ROI Modeling**: Template for calculating return on investment for framework adoption. Input variables (team size, project duration, current onboarding time) and projected savings with conservative and optimistic scenarios.
- **Cost of Not Adopting**: What happens without structured AI-assisted development -- inconsistent output, knowledge silos, repeated mistakes, slower onboarding. Concrete examples of waste the framework prevents.

## Diataxis Mapping

| Content Type | Title | Notes |
|-------------|-------|-------|
| Tutorial | "Try the framework in 30 min" | Hands-on walkthrough: clone, seed, design, first skill invocation |
| Explanation | Architecture overview | Why the framework is structured this way, design decisions |
| How-to | Adoption guide | Step-by-step: from evaluation to team rollout |
| Reference | Skills catalog | Complete list of skills with purpose, inputs, outputs |

## Tone Guidance

- Technical but accessible — assume engineering leadership vocabulary, avoid jargon that only framework authors would know.
- Honest about trade-offs — state clearly where the framework adds overhead or complexity, not just where it helps.
- Lead with problems solved — start from pain points the evaluator recognizes, then show how the framework addresses them.
- Respect the evaluator's time — front-load conclusions, put evidence in appendices.
