# USR — End Users

> People who use the software built with the framework.

## Core Question

"Does the software serve my needs well?"

## Roles

- Application end user
- Customer / Client user
- Internal business user
- Accessibility-dependent user
- Power user / Frequent user

## Communication Content

### Essential (all End User communications need)

- **Metacommunication Message**: Use the definition of metacommunication message in `general/shared-definitions.md`. If multiple user groups (roles, profiles etc.) are identified, create one metacommunication message for each. However, if there is a lot of overlap, indicate which parts (paragraphs?) of the metacommunication message are addressed to whom. In this latter case, consider starting with a metacommunication message that addresses all user groups, followed by sections addressed to each specific user group.
- **Quality Manifesto**: A plain-language commitment to the standards the development team holds. What users can expect in terms of reliability, usability, and responsiveness to feedback. Written from the perspective of people who build the software to the people who use it.
- **Feedback Channels**: How users can report issues, request features, and share their experience. Clear paths with expected response times. Emphasis on the team's commitment to listening.

### Deep-dive (load for thorough engagement or when End User trust is the primary goal)

- **Transparency Note**: How AI-assisted development is used in building the software. What it means for quality, what safeguards are in place, and what it does not mean (e.g., the software is not "written by AI" — it is designed, reviewed, and validated by people using AI as a tool).
- **Accessibility Statement**: The team's approach to accessibility — which standards are followed (WCAG level), how accessibility is tested, and how to report accessibility barriers.
- **Privacy Commitment**: How user data is handled during development (e.g., AI tools do not receive production user data). Clear statement on data practices relevant to the user.

## Diataxis Mapping

| Content Type | Title | Notes |
|-------------|-------|-------|
| Explanation | Quality manifesto | Why the team builds software the way they do |
| How-to | Feedback guide | How to submit feedback, report bugs, request features |

## Tone Guidance

- Non-technical — avoid developer jargon entirely. Write as if explaining to someone who has never written code.
- User-centric — focus on what the software does for the user, not how it is built. The framework is invisible to this audience.
- Warm but not patronizing — respect the user's intelligence while keeping language accessible.
- Focus on commitments — users want to know what they can rely on, not how the team organizes its work.
