# I18N — Internationalization

## Essential

- [P0] Are all user-facing strings using i18n keys (not hardcoded)?
- [P0] Are both locale files (pt-BR.json, en-US.json) and .po catalogs updated in sync?
- [P0] Are resource bundles structured so that new locales can be added without code changes, and are extraction/merge pipelines automated to prevent stale or orphaned keys? *(Localization engineer)*
- [P0] Are all text pipelines fully Unicode-safe (UTF-8 storage, NFC normalization, no truncation mid-codepoint), and are grapheme clusters handled correctly for operations like truncation, counting, and cursor movement? *(Unicode/encoding expert)*

## Deep-dive

- [P1] Are diacritics and special characters properly encoded?
- [P1] Are locale fallback chains tested (e.g., missing key in primary locale falls back gracefully)?
- [P1] Are date, number, and currency formats locale-aware in both backend responses and frontend rendering?
- [P1] Does the translation workflow integrate with a TMS (e.g., Crowdin, Phrase, Lokalise) via CI, ensuring source strings are pushed and translations pulled without manual file copying? *(Translation management system specialist)*
- [P1] Does the UI adapt to RTL scripts (Arabic, Hebrew) with correct mirroring of layouts, icons, and bidirectional text isolation (using `dir="auto"` or Unicode bidi controls) without breaking mixed-direction content? *(RTL layout specialist)*
- [P1] Are locale-sensitive formatters (ICU MessageFormat or Intl APIs) used for plurals, ordinals, relative time, unit formatting, and number grouping instead of hand-rolled string interpolation? *(Locale-aware date/number formatting expert)*
- [P2] Are culturally sensitive elements (colors, imagery, idioms, name/address field order, and humor) reviewed for target markets to avoid offense or confusion beyond mere linguistic translation? *(Cultural adaptation consultant)*
- [P2] Is there an automated QA gate (pseudo-localization, string-length expansion checks, screenshot diffing) that catches untranslated strings, clipped text, and concatenation-based sentences before release? *(Translation quality assurance analyst)*
- [P2] Do search indexes and sort collations use locale-aware rules (e.g., ICU collation, accent-insensitive matching) so that queries behave correctly across languages and scripts? *(Internationalized search/sort specialist)*
- [P2] Are internationalized interfaces accessible (correct `lang` attributes per content fragment, screen-reader-friendly locale switches, and sufficient contrast/spacing for complex scripts like Devanagari or CJK)? *(Accessibility + i18n intersection specialist)*
- [P3] Is there a defined content strategy for which locales receive full, partial, or machine-translated coverage, with clear metadata (locale completeness %, last-updated timestamps) exposed to stakeholders? *(Multilingual content strategist)*
