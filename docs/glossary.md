# Glossary

Key terms used throughout the SEJA framework documentation. Terms are listed alphabetically.

### Agent

A specialized subprocess that Claude Code can delegate tasks to. SEJA includes 10 agents: 7 evaluators (code-reviewer, plan-reviewer, advisory-reviewer, council-debate, standards-checker, test-runner, migration-validator) and 3 generators (communication-generator, onboarding-generator, document-generator).

**See also**: [Skills, Agents, and the Pipeline](concepts/skills-agents-pipeline.md)

### As-is

The current implementation state of the project, tracked in files like `conceptual-design-as-is.md` and `metacomm-as-is.md`. Updated automatically by post-skill after plan execution.

**See also**: [Design-Intent Lifecycle](concepts/design-intent-lifecycle.md)

### Brief

Your description of what you want done, given to a skill like `/plan` or `/advise`. A clear brief produces better plans. Not to be confused with the briefs log (`briefs.md`), which is the execution log of all skill invocations.

**See also**: [Skills, Agents, and the Pipeline](concepts/skills-agents-pipeline.md)

### CEM (Communicability Evaluation Method)

A method from semiotic engineering for evaluating how well an interface communicates the designer's intent. SEJA uses CEM-derived utterances (e.g., "Where is it?", "I can't do it this way") in the UX and DX review perspectives to identify communicative breakdowns.

**See also**: [Review Perspectives](concepts/review-perspectives-and-communicability.md)

### CDN (Cognitive Dimensions of Notations)

A framework for evaluating notations and APIs along 14 dimensions (viscosity, visibility, premature commitment, etc.). Used in SEJA's API review perspective.

**See also**: [Review Perspectives](concepts/review-perspectives-and-communicability.md)

### Communicability

The quality of a system's ability to communicate the designer's intent to users. A system with high communicability makes its logic, constraints, and available actions clear through its interface.

**See also**: [What Is SEJA](concepts/what-is-seja.md)

### Constitution

A project's immutable principles, stored in `project/constitution.md`. Generated by `/design`, never modified by agents. Defines non-negotiable quality, security, and design commitments.

**See also**: [Design-Intent Lifecycle](concepts/design-intent-lifecycle.md)

### Context budget

The strategy a skill uses to manage how much reference material is loaded into the LLM's context window. Three tiers: light (minimal), standard (default -- briefs index + eager refs), heavy (full briefs + all refs).

**See also**: [Context Budget and References](concepts/context-budget-and-references.md)

### Design intent

The target-state description of what the product should be, captured in `design-intent-to-be.md`. Includes both the domain model (entities, permissions) and the metacommunication vision (what the interface should communicate to users).

**See also**: [Design-Intent Lifecycle](concepts/design-intent-lifecycle.md)

### EMT (Extended Metacommunication Template)

A structured set of guiding questions from semiotic engineering that help designers articulate what they want the system to communicate at each lifecycle stage (analysis, design, prototype, evaluation).

**See also**: [What Is SEJA](concepts/what-is-seja.md)

### Established

The third layer of the design-intent lifecycle. Contains validated, processed design decisions that have been promoted from the as-is state by a human curator. Serves as the project's design history.

**See also**: [Design-Intent Lifecycle](concepts/design-intent-lifecycle.md)

### Lifecycle markers

Inline HTML comments that track the status of design-intent items. `<!-- STATUS: IMPLEMENTED | plan-NNNNNN | YYYY-MM-DD -->` marks an item as implemented. `<!-- ESTABLISHED: plan-NNNNNN | YYYY-MM-DD -->` marks promotion to the established archive.

**See also**: [Design-Intent Lifecycle](concepts/design-intent-lifecycle.md)

### Metacommunication

A message from the designer to the user, conveyed through the interface. In SEJA, metacommunication is always phrased as "I (designer) designed this for you (user) because...". The framework preserves this phrasing to maintain design intent.

**See also**: [What Is SEJA](concepts/what-is-seja.md)

### Pre-skill / Post-skill

Internal lifecycle hooks that run before and after every skill invocation. Pre-skill loads context (7 stages: help, brief-log, orphan-check, budget-eval, compaction-check, ref-load, constitution). Post-skill handles cleanup (brief update, as-is alignment, QA logging, commit, index regeneration).

**See also**: [Skills, Agents, and the Pipeline](concepts/skills-agents-pipeline.md)

### Reference

A Markdown file in `_references/` that encodes project standards, templates, or shared guidance. Three categories: general (universal), template (used by `/design`), project (generated per-project).

**See also**: [Context Budget and References](concepts/context-budget-and-references.md)

### Reflective practice

A concept from Donald Schon adopted by SEJA. Reflection-in-action (thinking while doing -- pre/post-skill hooks), reflection-on-action (reviewing what was done -- spec-drift, review perspectives), reflection-on-practice (questioning underlying assumptions -- behavior-evolution reconstruction).

**See also**: [What Is SEJA](concepts/what-is-seja.md)

### Review perspective

One of 16 domain-based evaluation lenses applied to code, plans, or decisions. 11 engineering perspectives (SEC, PERF, DB, API, ARCH, DX, I18N, TEST, OPS, COMPAT, DATA) and 5 design perspectives (UX, A11Y, VIS, RESP, MICRO).

**See also**: [Review Perspectives](concepts/review-perspectives-and-communicability.md)

### Sign classes

From semiotic engineering: static signs (always visible -- labels, icons), dynamic signs (change with interaction -- animations, state changes), and metalinguistic signs (explain other signs -- tooltips, help text, onboarding). Used in the UX review perspective.

**See also**: [Review Perspectives](concepts/review-perspectives-and-communicability.md)

### Skill

A reusable workflow invoked with `/skill-name` in Claude Code. Each skill is defined in a `SKILL.md` file with YAML frontmatter (following the agentskills.io spec) and Markdown instructions. SEJA includes 13 user-facing skills.

**See also**: [Skills, Agents, and the Pipeline](concepts/skills-agents-pipeline.md)

### To-be

The target-state design vision, captured in `design-intent-to-be.md`. Describes what the product should become, including entity hierarchy, permissions, UX patterns, and metacommunication intentions. Human-maintained; agents may propose changes but never edit directly.

**See also**: [Design-Intent Lifecycle](concepts/design-intent-lifecycle.md)
