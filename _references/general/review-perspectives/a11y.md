# A11Y — Accessibility

## Essential

- [P0] Does this meet WCAG AAA target requirements (contrast, keyboard navigation, screen readers)?
- [P0] Are interactive elements focusable with visible focus indicators?
- [P0] Are focus management and live region announcements tested with actual screen readers (NVDA, VoiceOver)?
- [P0] Are all interactive custom components (modals, dropdowns, pickers) fully keyboard-operable with documented shortcuts?
- [P0] Do dynamic content updates (SPAs, AJAX regions, toast notifications) produce correct screen reader announcements via aria-live regions, and have these been verified across NVDA, JAWS, and VoiceOver? *(Screen reader testing specialist)*
- [P0] Can every workflow be completed using only the keyboard, with no focus traps, logical tab order, and visible skip-navigation links? *(Keyboard-only navigation expert)*
- [P0] Does the product satisfy applicable legal accessibility mandates (ADA, Section 508, EN 301 549, EAA), and is there a published accessibility statement with conformance claims, known limitations, and a feedback mechanism? *(Legal / regulatory accessibility compliance officer)*

## Deep-dive

- [P1] Are ARIA labels and roles used where semantic HTML is insufficient?
- [P1] Do all color-dependent UI cues (errors, status, charts) provide a non-color alternative, and do palettes pass both WCAG AA and AAA contrast ratios including for large text, UI components, and graphical objects? *(Color/vision accessibility specialist)*
- [P1] Are instructions, error messages, and navigation patterns written in plain language, and are timeouts, animations, and multi-step processes designed to accommodate users with cognitive or learning disabilities? *(Cognitive accessibility advocate)*
- [P1] Has a formal WCAG 2.2 Level AAA conformance audit been performed, with all failures documented, prioritized, and tracked to resolution? *(WCAG auditor / compliance specialist)*
- [P1] Are ARIA roles, states, and properties used only when native HTML semantics are truly insufficient, and do complex composite widgets (grids, trees, tab panels) follow the corresponding WAI-ARIA Authoring Practices patterns? *(ARIA specialist)*
- [P1] Can all drag-and-drop, gesture-based, and hover-dependent interactions be performed through alternative single-pointer or switch-accessible controls with adequate target sizes (minimum 44x44 CSS pixels)? *(Motor impairment / switch access expert)*
- [P2] Are custom widgets built on top of well-supported platform primitives, and do they expose correct accessibility APIs (name, role, value, states) so that third-party assistive technologies can interact with them reliably? *(Assistive technology developer)*
- [P2] Are generated documents and PDFs tagged with correct structural markup (headings, lists, tables, reading order) and verified with automated checkers (PAC, Acrobat Accessibility Checker) to ensure assistive technology compatibility? *(Accessible document / PDF specialist)*
